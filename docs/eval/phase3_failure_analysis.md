### 5.2 Intro-parent 라벨 문제 (v2 구조 수정)

"`{isa}:{N}` = '용어의 정의는 다음과 같다.' 한 문장 intro" 인 청크에 primary 가 붙어 있으면 임베딩 신호가 사실상 0. 영향 쿼리:

|---|---|---|
| **Q14** | `200:13`(intro), 315:23, 330:5 | 200:13 → 대신 `200:(n)`(RMM 정의) 로 교체 |
| **Q16** | `200:13`(intro) 단독 | 대신 `200:(n)`,`200:(c)`,`200:(e)`,`200:(m)` 4개 서브 묶음 |
| **Q25** | `500:5`(intro) 단독 | 대신 `500:(b)`(적합성), `500:(c)`(감사증거), `500:(f)`(충분성) 3개 |

## 8. 역검증 로그 — v1 gold 관대 overmatch 전수 검토 (조건 B)

### 8.1 목적 및 방법

DA 가 제기한 "방향성 비대칭" 우려 해소. §5.1/§5.2 가 `gold 엄격 → primary 추가/교체` 로만 움직이면, 반대 방향의 "gold 가 관대해서 모델이 우연 맞춘 케이스" 가 숨겨질 수 있음. 이를 전수 검증.

- **대상**: `queries.yaml` v1 의 26 queries × 총 43 개 primary chunk_ids
- **방법**: 각 primary 를 `parsed/chunks.jsonl` 에서 chunk 로 해석 후 (positional resolver 포함), 실제 본문을 질의 의도와 수작업 대조. 판정 분류:
  - **(i) 정확 매칭**: primary 본문이 질의의 핵심 답변 — 이상적 gold.
  - **(ii) 프레이밍 미스매치(엄격 gold)**: primary 본문이 semantic 으로 부분 커버 or 프레이밍 불일치 — phase3 §5.3 에서 deferred 된 워크숍 대상.
  - **(iii) 관대 overmatch**: primary 본문이 질의 의도와 약하거나 무관하나 모델이 상위에 배치해 점수를 얻은 케이스 — **이번 검증의 타겟.**
  - **(iv) v2 에서 이미 수정**: §5 의 5건 (Q10/Q14/Q16/Q24/Q25) — 구조 이슈로 판단되어 본 검증 대상에서 분리.
- **점수 대조**: §3.2 실측(eval JSONs) 과 교차 — (iii) 후보는 모델이 top-5 안에 넣었으나 사람이 봤을 때 부적절한 경우.

### 8.4 (iii) 0건 판정의 근거

"관대 overmatch" 는 두 조건 동시 충족이 요구됨:
1. **sematic 부적절**: primary 본문이 질의 의도와 약하거나 무관.
2. **모델 통과**: 해당 쿼리에서 recall@5 ≥ 0.5 (즉, 실측에서 점수가 올라간 케이스).

전수 검토 결과:
- 실측 recall@5 ≥ 0.5 통과 쿼리(Q01/Q02/Q03(UPS)/Q04/Q05/Q06/Q07(UPS)/Q08/Q09/Q12(UPS)/Q13/Q15(UPS)/Q17/Q19/Q20/Q21/Q22/Q23/Q26 등)는 **모두 (i) 정확 매칭 또는 §5.3 이미 식별된 (ii) 프레이밍 미스매치**.
- (ii) 케이스(Q11/Q18/Q19-240:32 축)는 실측에서 모델이 gold 를 top-5 에 넣지 **못함** (recall@5 = 0.0 또는 해당 primary 특정 miss) — 즉 모델도 gold 가 엄격함을 확인함. "우연 맞춤" 성립 안 함.
- Q19 는 recall=0.5 통과지만 이는 `540:32` (i) 분이고 `240:32` (ii) 분은 miss. 절반 점수는 **정당한 절반**.

→ **관대 overmatch 0건 확정**. v2 수정의 "방향성 비대칭" 우려 해소. v2 는 false negative 제거 방향에만 작동했고 false positive 는 애초에 존재하지 않음.
